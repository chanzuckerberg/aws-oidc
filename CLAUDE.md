CGO_ENABLED=1 go test -race -coverprofile=coverage.txt -covermode=atomic ./...

go test ./pkg/okta/... -run TestGetClientIDs

The `scripts/rolemap/` directory has its own `go.mod` and is a standalone script — run it with `go run main.go` from that directory (requires `TFE_TOKEN` env var).

## Architecture

This is a CLI tool (`aws-oidc`) that exchanges OIDC tokens for temporary AWS credentials via `sts:AssumeRoleWithWebIdentity`. It also ships a webserver (`serve-config`) that serves personalized `~/.aws/config` content to authenticated users.

### Token flow

`execGetToken()` in `cmd/token.go` is the central auth function called by most commands. It uses `github.com/chanzuckerberg/go-misc/oidc/v5/cli` to obtain an OIDC ID token via either PKCE (browser redirect) or device code flow (`--device-code-flow`). Tokens are cached in `~/.cache/oidc-cli/` or a node-local path when `--node-local-cache`/`AWS_OIDC_NODE_LOCAL_CACHE` is set. `--flush-oidc-token-cache` clears the cache before proceeding.

### Client-side commands (`cmd/`)

- **creds-process** — Gets OIDC token, calls `getter.GetAWSAssumeIdentity()` (STS AssumeRoleWithWebIdentity), outputs JSON credentials. Used as a `credential_process` value in `~/.aws/config`.
- **exec** — Reads the current AWS profile from `~/.aws/config` (via `pkg/aws_config_client`), gets token, assumes role, sets `AWS_*` env vars, then `exec`s the given command.
- **configure** — Gets token, calls the `serve-config` endpoint (`pkg/aws_config_client`) to fetch available profiles, then uses `Survey` to interactively write `~/.aws/config` entries.
- **env** — Like `creds-process` but outputs shell `export` statements.
- **token** / **check-token-valid** / **check-refresh-ttl** — Token inspection utilities.

### Server (`serve-config`) architecture

The `serve-config` command starts an HTTP server (`pkg/aws_config_server`) with two responsibilities:

1. **AuthMiddleware** verifies the OIDC bearer token against the configured provider and injects the user's email and subject into the request context.
2. The root handler calls `pkg/okta` to list all Okta apps assigned to the user (paginated via `AppLister`), then looks each app's client ID up in the rolemap to produce a list of `AWSProfile` entries returned as YAML.

**Rolemap** (`/rolemap/rolemap.yaml` at runtime, `.infra/{rdev,prod}/rolemap/rolemap.yaml` in the repo): maps `okta_client_id` → `aws_role_arn` + account metadata. Auto-generated by `scripts/rolemap/main.go` from TFE state outputs and committed by the `update-rolemap` CI workflow on every push.

Environment variables required by `serve-config`: `OKTA_PRIVATE_KEY`, `OKTA_SERVICE_CLIENT_ID`, `OKTA_CLIENT_ID`, `OKTA_ISSUER_URL`.

### Key packages

- `pkg/getter` — STS AssumeRoleWithWebIdentity wrapper.
- `pkg/okta` — Okta SDK v3 client. `GetClientIDs()` paginates app listing filtered by user ID. `OIDCRoleMapping` / `OIDCRoleMappingsByKey` are the rolemap types.
- `pkg/aws_config_server` — HTTP router, auth middleware, config generation (`assemble_config.go`).
- `pkg/aws_config_client` — Reads `~/.aws/config` INI to extract `credential_process` parameters; also HTTP client for the serve-config endpoint.
